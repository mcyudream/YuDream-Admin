# Agent 模型工具调用与节点语义拆分设计

## 目标

将 Agent 工作流中的工具调用收归模型节点，使工具成为模型可自主选择的能力，而不是默认由独立节点直接执行。同时把当前笼统的节点目录拆成具有真实运行语义的输入、模型、知识、逻辑、转换和输出节点。

## 当前问题

- 应用级 `toolCodes`、独立 `tool` 节点和模型原生 Tool Calling 同时存在，工具授权、工具选择和工具执行职责重叠。
- `llm` 与 `understand` 只通过提示词区分用途，无法明确表达文本生成、结构化提取、意图分类和视觉理解。
- 自定义 Python 工具只能由独立 `tool` 节点执行，不能作为 Spring AI 原生 Tool Callback 提供给模型。
- 应用设置和节点设置都允许选择工具，保存时再临时合并，用户无法判断某个模型实际能调用哪些工具。

## 核心决策

### 模型节点拥有工具

每个聊天类模型节点持久化以下配置：

- `toolCodes: string[]`：该模型节点可见的系统工具和自定义 Python 工具。
- `toolMode: NONE | AUTO | REQUIRED`：禁止工具、模型自主选择、至少调用一次工具。
- 工具仍受应用授权、当前用户权限、插件生命周期和平台能力开关约束。

应用级 `AgentApplication.toolCodes` 保留为安全授权边界和反向查询字段，但不再由用户独立维护。前端保存工作流时取所有模型节点 `toolCodes` 的去重并集，后端再次从工作流计算并校验，避免客户端伪造授权。

### 模型节点实际语义拆分

新增或明确以下聊天模型节点语义：

- `llm`：文本生成，输出自然语言或 Markdown。
- `extract`：结构化提取，根据 `outputSchema` 返回 JSON；解析失败时节点失败。
- `classify`：意图分类，根据 `classes` 返回一个合法分类及可选原因。
- `vision`：视觉理解，要求选择支持 Vision 的聊天模型，并从运行输入或指定变量读取图片。
- `understand`：仅作为历史工作流兼容类型保留，新建节点面板不再展示；运行时按结构化提取处理。

Embedding 与 Rerank 仍属于模型目录，但不是聊天模型，不支持 Tool Calling。

### 节点目录

- 输入：开始、参数输入、文档解析。
- 模型：文本生成、结构化提取、意图分类、视觉理解、Embedding、Rerank。
- 知识：知识检索、向量检索、引用整理。
- 逻辑：条件分支、Python 代码。
- 转换：模板转换。
- 输出：结束输出。

独立“工具调用”节点从节点面板移除。确定性 Python 数据处理继续使用“Python 代码”节点，它不属于模型 Tool Calling。

## 运行时架构

### 工具解析

应用层新增可复用的 Agent 工具解析与执行组件：

1. 根据模型节点 `toolCodes` 查找系统 `AiAgentTool` 和已启用的自定义 `AgentTool`。
2. 校验工具属于应用授权并校验当前权限快照。
3. 将自定义 Python 工具适配为临时 `AiAgentTool`，描述、输入 Schema 和权限元数据来自工具记录。
4. 通过一次模型调用作用域把本节点的实际工具实例传给 AI 网关。
5. AI 网关只为当前作用域注册 Spring AI `FunctionToolCallback`，模型看不到其他应用或其他模型节点的工具。

系统工具和 Python 工具共用同一执行结果事件，继续通过 `AiAgentToolResult`、SSE `tool`/AG-UI 工具事件实时反馈。

### 模型调用

模型处理器按节点语义组装请求：

1. 解析节点模型、输入变量、附件和提示词。
2. 解析并授权节点工具。
3. 根据 `toolMode` 设置 Tool Calling 行为。
4. 执行流式模型调用和原生工具回调。
5. 按节点类型解析文本、JSON、分类或视觉结果。
6. 将结果写入 `outputVariable`，并发送节点完成或失败事件。

`REQUIRED` 模式在提供者原生支持时使用 Tool Choice；兼容提供者未执行任何工具时，模型节点以明确错误结束，不能静默当作成功。

## 兼容与迁移

- 后端继续注册旧 `tool` 节点处理器，保证已发布和插件内历史工作流可运行。
- 前端加载旧工作流时保留旧节点并显示“兼容节点”标识，但节点面板不能新增它。
- 若旧 `tool` 节点与唯一上游聊天模型直接相连，可在编辑器提供无损迁移：把工具加入上游模型 `toolCodes` 后移除旧节点并重连边。
- 无法确定模型归属的旧工具节点不自动迁移，避免改变确定性流程语义。
- 内置 CMS Agent 将 CMS 工具配置到实际执行构建的模型节点；澄清、分类节点不默认获得画布修改工具。
- 插件 Agent 工作流继续读取原 JSON；插件升级后可逐步采用模型节点 `toolCodes`。

## 前端交互

- 模型节点检查器增加“模型能力”“工具调用”“输入输出”三个独立区块。
- 工具使用可搜索多选列表，区分系统、Python 和插件工具，并显示权限或不可用原因。
- `toolMode=NONE` 时隐藏工具选择；`AUTO/REQUIRED` 且未选工具时阻止发布。
- 模型节点卡片显示模型名称、工具数量和工具模式；视觉节点额外显示图片输入状态。
- 应用设置不再展示第二套工具复选列表，只显示“由模型节点汇总的授权工具”只读摘要。

## 校验与错误处理

- 前后端校验器都验证模型类型、模型能力、工具存在性、工具启用状态、应用授权并集和工具模式。
- Vision 节点拒绝非视觉模型；运行时缺少图片时返回节点级错误。
- Extract 节点校验 `outputSchema` 是 JSON 对象；返回值不符合 JSON 时失败。
- Classify 节点要求至少两个不重复分类；返回未知分类时失败。
- Python 工具继续遵守 `def run(params: dict) -> dict`、超时和字典输出契约。
- 工具权限在异步执行前使用已捕获权限快照校验，不从异步线程读取登录上下文。
- 插件卸载或工具禁用后，模型节点发布校验失败，运行中的新调用也不能解析该工具。

## 测试策略

- 前端单元测试覆盖节点默认数据、旧数据归一化、工具并集、各模型节点校验和旧工具节点迁移。
- 应用层测试覆盖节点级工具隔离、系统工具回调、Python 工具回调、权限拒绝和 `REQUIRED` 未调用失败。
- 工作流校验测试覆盖 Extract Schema、Classify 分类、Vision 模型能力和工具不存在。
- 内置 Agent 初始化测试确认 CMS 工具只挂载在构建模型节点，并能升级旧内置工作流。
- 浏览器验证节点目录、模型工具多选、保存重载、调试工具事件和画布实时状态。

## 非目标

- 本次不实现循环、并行汇聚、人工审批或子工作流节点。
- 本次不改变工具 CRUD 页的 Python `run(dict) -> dict` 契约。
- 本次不允许普通聊天响应渲染 AG-UI 卡片；现有 AG-UI 专用约束保持不变。
